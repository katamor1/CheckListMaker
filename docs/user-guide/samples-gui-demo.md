# CheckListMaker samples GUIデモ利用手順書

- 対象実装: `main`のElectron / React版
- 対象アプリケーション: CheckListMaker 0.1.0
- 対象環境: Windows 11 x64
- 文書版: 1.0
- 改訂日: 2026-07-16

> **重要**  
> この手順は、`main`に実装済みの参考資料登録、参考資料ロール、チェックリスト編集、9条件フォーム、4スコープを使用します。旧手順書にあった「現行GUIではフルサンプルを再現できない」という制約は解消されています。ただし、安定版リリース前のため、生成したEXEは開発・受入試験用として扱ってください。

## 0. この手順書で確認すること

このデモでは、架空の「設備状態監視機能 基本設計レビュー」を題材に、次をGUIから設定します。

1. 既存DOCXを主対象としてレビューするプロジェクト
2. 生成指示からDOCXを作成してレビューするプロジェクト
3. 4件の参考資料と権威レベル／優先順位
4. 必須参考資料ロール1件
5. 5チェック項目、9条件、4種類の評価範囲
6. `.clmproj`保存、事前検査、Copilot用ZIP出力
7. Copilot上での自己テスト、評価、`result.json`生成

`samples`には完成済み`.clmproj`、`.clmcheck`、実行後の`result.json`を置きません。GUIの操作と自動採番を含めて確認するためです。

## 1. 使用するサンプルファイル

サンプルルートは次です。

```text
samples\ja-machine-control-design-review
```

| 用途 | ファイル |
|---|---|
| 既存文書モードの主対象 | `existing-document\target\basic-design-before-review.docx` |
| 文書生成モードの入力値 | `generation\document-request.json` |
| 説明用の期待結果 | `existing-document\expected-outcomes.json` |
| 拘束力のある品質規則 | `references\quality-assurance-policy.pdf` |
| 承認済み構成テンプレート | `references\basic-design-template.md` |
| 作業用語集 | `references\control-terminology.txt` |
| 参考レベルの記述例 | `references\reference-design.docx` |

`expected-outcomes.json`はCopilotへアップロードしません。正式な結果は実行後の`outputs/result.json`です。

## 2. アプリの準備と起動

### 2.1 開発モードで起動する

PowerShellで次を実行します。

```powershell
git clone https://github.com/katamor1/CheckListMaker.git
cd CheckListMaker
git checkout main
npm install --no-audit --no-fund
npm run dev
```

### 2.2 ポータブルEXEを作成する

```powershell
npm run dist:portable
.\artifacts\portable\CheckListMaker.exe
```

配布先でNode.jsやnpmを要求しない単一EXEは`artifacts\portable\CheckListMaker.exe`です。コード署名とWindows 11クリーン環境の受入が完了するまでは、組織外へ本番配布しないでください。

## 3. 現在のGUI構成

起動直後の上部には次のボタンがあります。

- `既存文書を検証`
- `文書を生成して検証`
- `プロジェクトを開く`

プロジェクト作成後、中央のワークスペースに次のタブが表示されます。

| タブ | 主な操作 |
|---|---|
| `概要・文書` | プロジェクト名、主対象文書、または文書生成設定 |
| `参考資料` | ファイル追加、表示名、用途、権威レベル、優先順位、有効日、ロール割当 |
| `チェックリスト` | チェックリスト、参考資料ロール、チェック項目、9条件、4スコープ |

画面下部には`保存`、`名前を付けて保存`、`事前検査`、`Copilot用ZIPを作成`があります。右側には事前検査のエラー／警告が表示されます。

## 4. IDを期待値どおりにするためのルール

IDはGUIが自動採番し、利用者は編集できません。このデモの期待結果は次のIDを前提にします。

- チェック項目: `CHK-0001`～`CHK-0005`
- 条件: `COND-01`～`COND-09`
- 参考資料: `REF-001`～`REF-004`
- 参考資料ロール: `ROLE-001`

条件IDを一致させるには、次の順序を守ります。

| 操作順 | 結果 |
|---:|---|
| 1 | 新規プロジェクトの既定項目が`CHK-0001`、既定条件が`COND-01` |
| 2 | `CHK-0001`へ条件を追加して`COND-02` |
| 3 | チェック項目を追加して`CHK-0002`／`COND-03` |
| 4 | `CHK-0002`へ条件を追加して`COND-04` |
| 5 | チェック項目を追加して`CHK-0003`／`COND-05` |
| 6 | `CHK-0003`へ条件を追加して`COND-06` |
| 7 | チェック項目を追加して`CHK-0004`／`COND-07` |
| 8 | `CHK-0004`へ条件を追加して`COND-08` |
| 9 | チェック項目を追加して`CHK-0005`／`COND-09` |

条件や項目を削除すると、そのIDは廃止され再利用されません。手順途中でIDがずれた場合は、期待結果を使うデモでは新規プロジェクトからやり直すのが確実です。

参考資料は複数選択できますが、Windowsのファイル選択順に依存しないよう、`REF-001`から`REF-004`まで1件ずつ追加します。

## 7. 事前検査、保存、ZIP出力

1. 右側の件数が更新されていることを確認します。
2. `事前検査`をクリックします。
3. エラーがある場合は、指摘メッセージと対処を確認して設定を修正します。
4. 警告は意図した設定か確認します。たとえば必須項目で`対象外を許可`をオンにすると警告になります。
5. `名前を付けて保存`をクリックし、`設備状態監視機能_既存文書デモ.clmproj`として保存します。
6. 一度`プロジェクトを開く`から保存した`.clmproj`を開き、主対象、4参考資料、5項目、9条件が復元されることを確認します。
7. `Copilot用ZIPを作成`をクリックします。
8. `設備状態監視機能_既存文書デモ-copilot-package.zip`として保存します。
9. フッターの`生成したZIPを表示`をクリックし、出力場所を開きます。

## 8. 文書生成モードのデモ

既存文書プロジェクトとは別に新規作成します。

### 8.1 文書生成設定

1. `文書を生成して検証`をクリックします。
2. プロジェクト名を`設備状態監視機能 基本設計レビュー（文書生成）`にします。
3. `概要・文書`タブで次を入力します。

| GUI項目 | 入力値 |
|---|---|
| 文書タイトル | 設備状態監視機能 基本設計書 |
| 想定読者 | 制御ソフトウェア設計者および品質保証担当者 |
| 文書の目的 | 承認レビュー用の基本設計書初稿を作成する |
| 言語 | ja |
| 生成形式 | Word (.docx) |
| 文書生成指示 | 参考資料を事実根拠として使用し、根拠のない値を創作せず、「1. 目的」「2. 適用範囲」「3. 構成」「4. 機能設計」「5. 異常処理」「6. スケジュール」「7. 承認」の七つの必須セクションをこの順序で出力する。 |
| 参考資料を事実の根拠として使用する | オン |
| 参考資料にない事実を推測で補わない | オン |

4. 第5章と第6章と同じ順序で、参考資料ロール、4参考資料、5チェック項目、9条件を設定します。
5. `事前検査`を実行します。
6. `設備状態監視機能_文書生成デモ.clmproj`として保存します。
7. Copilot用ZIPを出力します。

## 9. Copilotで実行する

1. 生成したZIPを、アップロード済みファイルを読み取れてPythonを実行できるブラウザ版Copilotへアップロードします。
2. ZIP内の`00_READ_ME_FIRST.md`を確認します。
3. `01_EXECUTION_PROMPT.md`の内容をCopilotへ送信します。
4. Copilotが最初に次を実行したことを確認します。

```bash
python validate_output.py --self-test
```

5. Copilotに、対象文書の評価、許可範囲の修正、再評価、`result.draft.json`の作成を続行させます。
6. Copilotがバリデータを実行し、合格後に`outputs/result.json`を確定したことを確認します。
7. `outputs/result.json`、人間向けレポート、修正文書または生成文書、検証記録をダウンロードします。
8. `result.json`を唯一の正本として扱います。

| `COND-08` | `valid` | `社内`が許可値に含まれる |
| `COND-09` | `invalid` | `binding`の250 ms以下を`reference`の500 msより優先 |

`CHK-0005`は任意項目なので、`COND-09`の不適合は全体判定では警告として扱われます。`REF-001`の品質保証規程が`REF-004`の参考設計書より優先されたことを、根拠と資料解決記録で確認します。

## 11. トラブルシューティング

| 症状 | 原因と対処 |
|---|---|
| 条件IDが期待値と違う | 項目・条件の追加順が異なります。削除済みIDは再利用されないため、新規プロジェクトで第4章の順序どおり作り直します。 |
| 参考資料IDが期待値と違う | 4ファイルを一括選択した可能性があります。新規プロジェクトでREF-001から1件ずつ追加します。 |
| `品質基準`チェックが表示されない | 先に`チェックリスト`タブでROLE-001を追加します。 |
| 参考資料整合性で選択肢がない | 先に`参考資料`タブで4件を登録します。 |
| `参考資料 REF-xxx が存在しません` | 条件で選択した資料を削除しています。COND-09の選択を更新します。 |
| 主対象文書がありません | `概要・文書`タブで`文書を選択`を実行します。 |
| 文書生成指示が空です | 第8章の指示文を入力します。JSONではキー名が`instructions`です。 |
| チェック項目に条件がありません | 削除した条件の代わりに1件以上追加します。 |
| 保存またはZIP作成ができない | 事前検査のエラーをすべて解消します。 |
| PDFを自動修正できない | PDFは評価・参照専用です。修正案のみ、または変更禁止を使用します。 |
| Copilot結果が期待と違う | ZIP内の`checklist.json`、`references/`、`manifest.json`と、実際の条件IDを確認します。`expected-outcomes.json`はAIへ渡しません。 |
