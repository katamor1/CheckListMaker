# Electron版 日本語UI文言設計

- 文書状態: 会話上の設計承認済み・書面レビュー待ち
- 作成日: 2026-07-16
- 対象リポジトリ: `katamor1/CheckListMaker`
- 対象ブランチ: `agent/japanese-ui-copy`
- 対象アプリケーション: Electron版 CheckListMaker

## 1. 目的

Electron版CheckListMakerで利用者が目にする文言を、日本人の業務利用者を前提として全面的に整理する。

本変更は、用語を過度に平易化するものではない。正確さを優先した専門的な日本語を維持しながら、次の問題を解消する。

- 英語見出しと日本語本文が混在している
- 同じ操作や状態に複数の表現が使われている
- ボタン名だけでは操作結果を判断しにくい
- 未設定状態が「なし」「未選択」だけで示され、何が不足しているか分かりにくい
- 事前検査のエラーコードが通常利用者へ常時表示されている
- 内部例外文がそのまま利用者へ表示される可能性がある
- React画面とElectron Main Processのダイアログで用語が一致していない

## 2. 対象利用者

主な対象は、次の条件を満たす日本語利用者である。

- Windows 11を業務で使用する
- 文書レビューや要件確認を担当する
- JSON、Python、Electron、IPCなどの内部技術を理解しているとは限らない
- `プロジェクト`、`テンプレート`、`チェックリスト`、`Copilot`、`ZIP`、`JSON`、`Python`、`DOCX`、`PDF`など、定着した技術用語は理解できる
- エラー発生時には、何が起きたか、データが安全か、次に何をすべきかを知る必要がある

## 3. 基本方針

### 3.1 正確さ優先

専門用語は無理に日本語へ置き換えない。

次の語はそのまま使用する。

```text
プロジェクト
テンプレート
チェックリスト
Copilot
ZIP
JSON
Python
DOCX
PDF
```

ただし、内部実装の都合だけで使われている用語は画面へ出さない。

例:

- `Renderer`、`IPC`、`dirty`、`preflight`は通常表示しない
- `事前検査`、`未保存の変更あり`など、利用者の作業に対応する日本語を表示する

### 3.2 英語見出しを使用しない

画面上の装飾見出しを含め、利用者向け見出しは日本語へ統一する。

| 変更前 | 変更後 |
|---|---|
| `PROJECT` | `プロジェクト` |
| `PREFLIGHT` | `事前検査` |
| `LOCAL DOCUMENT VALIDATION PACKAGE BUILDER` | `文書検証パッケージ作成ツール` |

内部ID、ファイル名、コマンド、エラーコードは英数字のまま保持する。

### 3.3 操作名と通知の文体

ボタン、メニュー、リンクなどの操作名は、短く具体的な表現にする。句点は付けない。

結果通知、警告、エラー、説明文は丁寧語で統一する。

例:

```text
操作名: 事前検査を実行
通知: 事前検査が完了しました。

操作名: 上書き保存
通知: プロジェクトを保存しました。
```

### 3.4 表記規則

- 日本語と英数字の間へ不要な空白を入れない
- 三点リーダーは`…`を使う
- 日本語文章の括弧は全角`（ ）`を基本とする
- ファイル名、コマンド、内部ID、エラーコードは等幅表示またはバッククォートで示す
- `下さい`ではなく`ください`を使う
- `出来ます`ではなく`できます`を使う
- `行なう`ではなく`行う`を使う
- 同じ概念には必ず同じ語を使う

## 4. 文言管理アーキテクチャ

### 4.1 配置

利用者向けの共通文言を、次のモジュールへ集約する。

```text
src/shared/presentation/ja/
├── terminology.ts
├── actions.ts
├── statuses.ts
├── messages.ts
├── dialogs.ts
├── validation-messages.ts
└── index.ts
```

### 4.2 各ファイルの責務

#### `terminology.ts`

製品全体で使う基本用語を定義する。

例:

```typescript
export const terminology = {
  project: 'プロジェクト',
  checklist: 'チェックリスト',
  referenceDocument: '参考資料',
  targetDocument: '主対象文書',
  generationSettings: '文書生成設定',
  preflight: '事前検査'
} as const;
```

#### `actions.ts`

ボタン、リンク、メニューの操作名を定義する。

例:

```typescript
export const actions = {
  createExistingProject: '既存文書の検証を開始',
  createGenerationProject: '文書生成と検証を開始',
  openProject: 'プロジェクトを開く',
  saveOverwrite: '上書き保存',
  saveAs: '名前を付けて保存',
  selectTargetDocument: '主対象文書を選択',
  runPreflight: '事前検査を実行',
  createCopilotPackage: 'Copilot用ZIPを作成',
  openExportLocation: '生成したZIPの保存場所を開く',
  showDetails: '詳細を表示',
  hideDetails: '詳細を閉じる'
} as const;
```

#### `statuses.ts`

状態、モード、判定結果の表示名を定義する。

例:

```typescript
export const statuses = {
  saved: '保存済み',
  unsaved: '未保存の変更あり',
  processing: '処理中',
  existingDocumentMode: '既存文書を検証',
  documentGenerationMode: '文書を生成して検証'
} as const;
```

#### `messages.ts`

完了通知、空状態、警告、一般エラーを定義する。

同じ操作の成功・キャンセル・失敗を近くに配置し、表記の対応関係を保つ。

#### `dialogs.ts`

Electronのファイル選択・保存・確認ダイアログで使うタイトル、フィルター名、確認文を定義する。

#### `validation-messages.ts`

事前検査コードに対応する利用者向け見出しと対処方法を定義する。

内部コードそのものは変更しない。

#### `index.ts`

ReactとElectron Main Processが利用する公開入口とする。各コンポーネントが個別ファイルの内部構造へ依存しないようにする。

### 4.3 集約対象

次の文言は必ず共通モジュールから取得する。

- 基本用語
- ボタン、メニュー、リンク
- 作業モード名
- 保存、処理、成功、失敗、キャンセルの状態
- ファイルダイアログのタイトルとフィルター名
- 事前検査の見出し、メッセージ、対処方法
- 主要な空状態
- アクセシビリティ用ラベル

画面固有で一度しか使わない長い説明文は、無理に共通化しなくてよい。ただし本設計の文体・表記規則に従う。

### 4.4 導入しないもの

MVPは日本語専用であるため、翻訳ライブラリや汎用i18n基盤は導入しない。

次は行わない。

- `react-i18next`などの追加
- すべての文言を機械的な翻訳キーへ変換
- 英語リソースの同時作成
- OS言語による表示切替

### 5.5 空状態・未設定状態

短い否定語だけで済ませず、不足している対象を明示する。

```text
プロジェクトが開かれていません。
```

## 6. 通知メッセージ

### 6.1 成功

```text
既存文書を検証するプロジェクトを作成しました。
文書を生成して検証するプロジェクトを作成しました。
プロジェクトを開きました。
プロジェクトを保存しました。
プロジェクトに名前を付けて保存しました。
主対象文書を登録しました。
参考資料を登録しました。
事前検査が完了しました。
事前検査に合格しました。
Copilot用ZIPを作成しました。
チェックリストテンプレートを保存しました。
チェックリストテンプレートを開きました。
```

成功通知でファイル数などの補助情報を示す場合も、文を崩さない。

例:

```text
Copilot用ZIPを作成しました。パッケージには12ファイルが含まれています。
```

### 6.2 キャンセル

利用者がダイアログを閉じた操作は、原則としてエラー扱いしない。

必要な場合だけ次のように表示する。

```text
保存をキャンセルしました。
ファイルの選択をキャンセルしました。
```

### 6.3 失敗

エラーは次の順序で構成する。

1. 何ができなかったか
2. データが変更されたか、元ファイルが安全か
3. 利用者が次に確認すること
4. 必要な場合だけ詳細コード

例:

```text
プロジェクトを保存できませんでした。
元のプロジェクトファイルは変更されていません。
保存先への書き込み権限と、ファイルがほかのアプリで開かれていないか確認してください。
```

```text
主対象文書を読み込めませんでした。
元の文書は変更されていません。
ファイルが移動、削除、または破損していないか確認してください。
```

予期しないエラーは次の表現を基本とする。

```text
処理中に予期しない問題が発生しました。
元のファイルは変更されていません。
アプリを再起動して、もう一度操作してください。
```

## 7. 内部エラーの変換

Node.js、Electron、OS、ZIP処理などの内部例外文を、そのまま画面へ表示しない。

Main Processまたは共通のエラー変換層で、安定した利用者向けメッセージへ変換する。

### 7.1 変換結果

利用者向けエラーは、少なくとも次を保持できる構造にする。

```typescript
export interface UserFacingError {
  title: string;
  message: string;
  dataSafety?: string;
  nextAction?: string;
  detailCode?: string;
}
```

`detailCode`は通常表示しない。

### 7.2 代表的な変換

| 内部状況 | 利用者向け見出し | 次の操作 |
|---|---|---|
| ファイル不存在 | `ファイルを読み込めませんでした。` | 移動・削除を確認する |
| アクセス拒否 | `ファイルへアクセスできませんでした。` | 権限と使用中状態を確認する |
| ZIP破損 | `プロジェクトファイルを開けませんでした。` | 正しい`.clmproj`か確認する |
| ハッシュ不一致 | `ファイルの内容を確認できませんでした。` | 元ファイルを選択し直す |
| 保存失敗 | `プロジェクトを保存できませんでした。` | 保存先と使用中状態を確認する |
| パッケージ生成失敗 | `Copilot用ZIPを作成できませんでした。` | 事前検査と保存先を確認する |

開発者向けの生の例外情報は、将来のログ機能へ渡せるよう保持してよいが、利用者画面には表示しない。

## 8. 事前検査の表示

### 8.1 通常表示

事前検査の一覧では、日本語の見出しと対処方法を主情報として表示する。

```text
主対象文書が選択されていません。
MD、TXT、DOCX、またはPDFの主対象文書を選択してください。

[詳細を表示]
```

エラーコードを常時表示しない。

### 8.2 詳細表示

`詳細を表示`を選択したときだけ、調査用情報を表示する。

```text
エラーコード: TARGET_REQUIRED
対象画面: 主対象文書
対象項目: target
```

詳細表示は項目ごとに開閉できる。

### 8.3 件数表示

件数表示は次に統一する。

```text
エラー 2件、警告 1件
```

0件の場合も省略せず表示してよい。

```text
エラー 0件、警告 0件
```

### 8.4 検査前の空状態

```text
事前検査はまだ実行されていません。
「事前検査を実行」を選ぶと、保存やCopilot用ZIPの作成前に入力内容を確認できます。
```

## 9. Electronダイアログ

### 9.1 統一表

| 用途 | ダイアログタイトル | フィルター表示 |
|---|---|---|
| プロジェクト読込 | `プロジェクトを開く` | `CheckListMakerプロジェクト（.clmproj）` |
| プロジェクト保存 | `プロジェクトに名前を付けて保存` | `CheckListMakerプロジェクト（.clmproj）` |
| 主対象文書 | `主対象文書を選択` | `対応文書（MD、TXT、DOCX、PDF）` |
| 参考資料 | `参考資料を選択` | `対応文書（MD、TXT、DOCX、PDF）` |
| テンプレート読込 | `チェックリストテンプレートを開く` | `チェックリストテンプレート（.clmcheck）` |
| テンプレート保存 | `チェックリストテンプレートに名前を付けて保存` | `チェックリストテンプレート（.clmcheck）` |

### 9.2 ファイル名の既定値

既定ファイル名は、現在のプロジェクト名またはチェックリスト名を使う。空の場合のフォールバックも日本語で統一する。

```text
新しいプロジェクト.clmproj
新しいチェックリスト.clmcheck
新しいプロジェクト-copilot-package.zip
```

パッケージ内部のASCIIファイル名規則は変更しない。

## 10. アクセシビリティ文言

`aria-label`、`aria-live`、見出し、ボタンの読み上げ内容も共通文言と同じ語彙を使用する。

次を満たす。

- 表示ラベルと読み上げラベルが矛盾しない
- アイコンだけの操作には明示的な日本語名を設定する
- 状態を色だけで示さず、`保存済み`、`未保存の変更あり`などの文字を併記する
- エラーと警告を読み上げ可能にする
- 処理開始・完了・失敗を`aria-live`で通知する
- `詳細を表示`と`詳細を閉じる`の状態を読み上げられるようにする

## 11. 適用範囲

### 11.1 対象

#### React Renderer

- 画面タイトル
- セクション見出し
- ボタン、リンク、メニュー
- 入力項目名
- 補足説明
- 状態表示
- 空状態、未設定状態
- 事前検査結果
- 完了通知、警告、エラー
- `aria-label`などのアクセシビリティ文言

#### Electron Main Process

- ファイル選択ダイアログ
- 保存ダイアログ
- ファイル形式フィルター
- 読込、保存、エクスポート時のエラー
- 起動失敗時のメッセージ
- システム処理結果をRendererへ返す利用者向け文言

#### Preload

Preload自体で表示文言を生成しない。Main Processから返された構造化情報を、そのままRendererへ安全に渡す。

### 11.2 対象外

次は変更しない。

- `project.json`、`checklist.json`などのプロパティ名
- `existing_document`などの内部列挙値
- `TARGET_REQUIRED`などのエラーコード
- `.clmproj`、`.clmcheck`、Copilot用ZIPの形式
- パッケージ内部のファイル名
- `00_READ_ME_FIRST.md`
- `01_EXECUTION_PROMPT.md`
- `02_CONTINUE_PROMPT.md`
- Pythonバリデータ内の文言
- サンプル文書本文
- サンプルの期待結果本文

本変更によって、既存プロジェクト、テンプレート、Copilot用ZIPとの互換性を変更しない。

## 12. テスト方針

### 12.1 文言契約テスト

共通文言モジュールに対し、主要な用語と操作名を固定するテストを追加する。

例:

- `PROJECT`、`PREFLIGHT`などの英語見出しが公開文言に含まれない
- 上書き保存の操作名が`上書き保存`である
- 状態名が`未保存の変更あり`と`保存済み`である
- 技術用語`Copilot`、`ZIP`、`JSON`、`Python`を意図せず翻訳していない

### 12.2 Rendererテスト

主要画面について次を確認する。

- 英語装飾見出しが表示されない
- ボタン名が統一表と一致する
- 空状態が対象を明示する文になっている
- 事前検査コードが初期状態で非表示である
- `詳細を表示`でコード、画面、項目が表示される
- `詳細を閉じる`で再び非表示になる
- 保存状態と処理状態が文字で表示される
- 文書生成フォームの項目名が統一されている

### 12.3 Main Processテスト

ダイアログ設定を純粋関数または定数として検査できるようにする。

次を確認する。

- ダイアログタイトルが統一表と一致する
- フィルター表示に拡張子が含まれる
- 選択キャンセルをエラーとして扱わない
- OS由来の生のエラー文をRendererへ直接返さない
- 利用者向けエラーに見出し、説明、データ安全性、次の操作が含まれる

### 12.4 互換性テスト

- JSON契約のfixtureが変わらない
- `.clmproj`、`.clmcheck`のround-trip結果が変わらない
- Copilot用ZIPの構造とハッシュ契約が、表示文言変更の影響を受けない
- サンプルカタログ検証が引き続き成功する

### 12.5 目視確認

Windows上のElectron画面で次を確認する。

- 日本語見出しが自然に読める
- ボタンが折り返されても意味が失われない
- 100%、150%、200%表示倍率で文言が欠けない
- 長いエラーメッセージがスクロール可能で、操作ボタンを隠さない
- ファイルダイアログのタイトルとフィルターが統一されている
- キーボードだけで詳細表示を開閉できる

## 13. 受入基準

次をすべて満たした場合に実装完了とする。

1. 利用者向けの英語装飾見出しが残っていない。
2. 基本用語、操作名、状態名が共通モジュールから取得されている。
3. 同じ意味の操作に複数の表示名が使われていない。
4. ボタンは簡潔な操作名、通知と説明は丁寧語で統一されている。
5. 空状態が不足対象を明示している。
6. 事前検査コードは通常非表示で、詳細表示時だけ確認できる。
7. ファイル選択・保存ダイアログのタイトルとフィルターが統一されている。
8. OSやNode.js由来の生の例外文が通常画面へ直接表示されない。
9. エラーに、失敗内容、データ安全性、次の操作が含まれる。
10. 表示ラベルとアクセシビリティ文言が一致している。
